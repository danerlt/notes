# 向量数据库概述

## 什么是向量数据库

向量数据库（Vector Database）是一种专门为存储和检索高维向量而设计的数据库。它的核心能力是**语义相似度搜索**：给定一个查询向量，快速找出数据库中与之最相似的若干个向量。

传统数据库存储的是结构化数据（整数、字符串、日期等），通过精确匹配或范围查询来检索。而向量数据库存储的是**嵌入向量（Embedding）**——由机器学习模型生成的浮点数数组，代表文本、图像、音频等非结构化数据的语义信息。

```
文本 "苹果"  ──► Embedding 模型 ──► [0.12, -0.34, 0.87, ..., 0.05]  (1536维)
图片 🍎      ──► Embedding 模型 ──► [0.11, -0.31, 0.89, ..., 0.07]  (1536维)
                                              ↑
                                    两者在向量空间中距离很近
```

## 为什么需要向量数据库

### 传统数据库的局限性

| 能力 | 关系型数据库（MySQL等） | 向量数据库 |
|------|------------------------|------------|
| 精确匹配查询 | 极佳 | 一般 |
| 范围查询 | 极佳 | 一般 |
| 语义相似度查询 | 不支持 | 核心能力 |
| 高维数据索引 | 极差（维度灾难） | 专项优化 |
| 亿级向量检索速度 | 极慢（全量扫描） | 毫秒级 |

**维度灾难**：当数据维度增高（如 1536 维），传统索引（B-Tree、Hash）会完全失效，只能做全量线性扫描，在亿级数据量下延迟无法接受。向量数据库使用专门的 ANN（近似最近邻）算法来解决这个问题。

### 核心价值

1. **语义理解**：不依赖关键词精确匹配，理解"苹果手机"和"iPhone"的语义相似性
2. **多模态支持**：文本、图片、音频、视频都可以转换为向量统一检索
3. **高性能 ANN 搜索**：毫秒级返回亿级数据中最相似的 Top-K 结果
4. **可扩展性**：支持水平扩展以应对大规模数据

## 核心概念

### 向量（Vector / Embedding）

向量是机器学习模型对数据语义的数值表示，本质是一个浮点数列表：

```python
# 使用 OpenAI Embedding API 生成向量
from openai import OpenAI

client = OpenAI()

def get_embedding(text: str) -> list[float]:
    response = client.embeddings.create(
        input=text,
        model="text-embedding-3-small"
    )
    return response.data[0].embedding

vec = get_embedding("机器学习是人工智能的一个分支")
print(f"向量维度: {len(vec)}")   # 1536
print(f"前5个分量: {vec[:5]}")   # [0.023, -0.045, 0.112, ...]
```

**常见 Embedding 模型及维度**：

| 模型 | 维度 | 特点 |
|------|------|------|
| OpenAI text-embedding-3-small | 1536 | 性价比高，速度快 |
| OpenAI text-embedding-3-large | 3072 | 精度更高 |
| sentence-transformers/all-MiniLM-L6-v2 | 384 | 开源，速度快 |
| BAAI/bge-large-zh-v1.5 | 1024 | 中文效果优秀 |
| nomic-embed-text | 768 | 开源，综合性能好 |

### 相似度度量

向量间的相似度通过距离函数计算：

```python
import numpy as np

def cosine_similarity(a: list, b: list) -> float:
    """余弦相似度：衡量两向量方向的相似程度，范围 [-1, 1]，越接近1越相似"""
    a, b = np.array(a), np.array(b)
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

def euclidean_distance(a: list, b: list) -> float:
    """欧式距离（L2）：两点之间的直线距离，值越小越相似"""
    a, b = np.array(a), np.array(b)
    return np.linalg.norm(a - b)

def dot_product(a: list, b: list) -> float:
    """点积（内积）：适用于已归一化的向量，计算最快"""
    return np.dot(np.array(a), np.array(b))

# 示例
v1 = [0.1, 0.5, 0.3]
v2 = [0.15, 0.48, 0.32]
v3 = [-0.9, 0.1, 0.2]

print(f"v1 vs v2 余弦相似度: {cosine_similarity(v1, v2):.4f}")  # 接近 1.0
print(f"v1 vs v3 余弦相似度: {cosine_similarity(v1, v3):.4f}")  # 较低
```

**选择建议**：
- **余弦相似度（Cosine）**：最常用，适合文本语义相似，对向量长度不敏感
- **欧式距离（L2）**：适合图像特征等绝对位置重要的场景
- **点积（IP / Inner Product）**：向量已归一化时等价于余弦相似度，计算更快

### 索引算法（ANN）

向量数据库不做精确最近邻搜索（太慢），而使用**近似最近邻（ANN）**算法，在速度和精度间取得平衡：

| 算法 | 全称 | 特点 |
|------|------|------|
| HNSW | Hierarchical Navigable Small World | 查询速度最快，内存占用大，推荐首选 |
| IVF_FLAT | Inverted File + Flat | 内存较小，精度高，构建速度快 |
| IVF_PQ | IVF + Product Quantization | 压缩存储，内存极小，精度略降 |
| ScaNN | Scalable Nearest Neighbors | Google 出品，大规模场景优秀 |
| Annoy | Approximate Nearest Neighbors Oh Yeah | Spotify 出品，只读场景适用 |

**HNSW 简要原理**：

```
构建多层图结构：
  顶层（稀疏）:  A ---- C ---- F
  中层（中等）:  A -- B -- C -- E -- F
  底层（密集）:  A-B-C-D-E-F-G-H-I-J

查询时：
  1. 从顶层入口点出发，贪心搜索
  2. 逐层下降，范围收窄
  3. 在底层找到最终 Top-K 候选
```

### 各数据库简介

**Chroma**
- 定位：开发者友好的轻量级向量数据库
- 适用：原型开发、中小规模应用（百万级以内）、本地测试
- 亮点：零配置内嵌运行，LangChain / LlamaIndex 原生支持，API 极简

**Qdrant**
- 定位：高性能、功能丰富的生产级向量数据库
- 适用：需要强过滤能力的生产应用，千万到亿级规模
- 亮点：Rust 实现，性能卓越；Payload Filter 功能强大；同时支持 REST 和 gRPC

**Milvus**
- 定位：企业级分布式向量数据库
- 适用：超大规模（亿级+）向量存储与检索
- 亮点：支持多种索引类型、分区管理、高可用；有托管版 Zilliz Cloud

**Weaviate**
- 定位：语义搜索引擎，内置模块化 ML 集成
- 适用：需要内置 Vectorizer 模块自动向量化的场景
- 亮点：GraphQL API，内置文本/图像向量化模块，支持混合搜索

**Pinecone**
- 定位：全托管云向量数据库
- 适用：不想运维基础设施、快速上线生产
- 亮点：零运维，弹性扩展，Serverless 定价，有免费套餐

**pgvector**
- 定位：PostgreSQL 的向量扩展插件
- 适用：已有 PostgreSQL 业务系统，需要融合向量检索
- 亮点：沿用完整 PostgreSQL 生态，向量与业务数据同库，JOIN 查询

## 核心操作

### 基本 CRUD 操作模式

所有向量数据库都支持以下核心操作，概念上高度一致：

```python
# 以下是通用伪代码，展示跨数据库的操作模式

# 1. 创建集合（Collection / Index / Table）
db.create_collection(
    name="my_docs",
    dimension=1536,          # 向量维度，必须与 Embedding 模型输出一致
    distance="cosine"        # 相似度度量方式：cosine / l2 / ip
)

# 2. 插入向量（带元数据/Payload）
db.insert(
    collection="my_docs",
    vectors=[[0.1, 0.2, ...], [0.3, 0.4, ...]],
    metadata=[
        {"source": "doc1.pdf", "page": 1, "text": "原始文本内容"},
        {"source": "doc2.pdf", "page": 3, "text": "另一段文本内容"},
    ]
)

# 3. 相似度查询（Top-K）
results = db.query(
    collection="my_docs",
    query_vector=[0.1, 0.2, ...],   # 查询向量（查询文本的 Embedding）
    top_k=5,                         # 返回最相似的5条
    filter={"source": "doc1.pdf"}    # 元数据过滤（可选）
)

# 4. 删除
db.delete(collection="my_docs", ids=["id_1", "id_2"])

# 5. 更新（大多数数据库通过 upsert 实现）
db.upsert(
    collection="my_docs",
    id="id_1",
    vector=[0.15, 0.25, ...],
    metadata={"source": "doc1.pdf", "page": 1, "text": "更新后的文本"}
)
```

## 应用场景

### RAG（检索增强生成）

最主流的应用场景，为 LLM 提供实时外部知识，减少幻觉：

```
用户提问："我们公司的退款政策是什么？"
       │
       ▼
将问题转为查询向量（Embedding）
       │
       ▼
向量数据库检索相关文档段落（Top-K）
       │    → 检索到：退款政策第3条、客服手册第7页...
       ▼
将检索到的段落 + 原始问题 组合成 Prompt
       │
       ▼
LLM 基于上下文生成准确回答（有事实依据，不瞎编）
```

### 语义搜索

替代传统关键词搜索，真正理解用户意图：
- 搜索"性价比高的手机" → 找到描述中有"价格实惠""配置不错"的商品
- 搜索"心情不好" → 推荐带有"释放压力""放松心情"标签的内容
- 搜索"如何提升代码质量" → 匹配"重构最佳实践""代码审查指南"等文档

### 推荐系统

基于用户行为生成用户偏好向量，检索相似内容：
- 用户喜欢的文章 → 生成用户兴趣向量 → 找到最相似内容进行推荐
- 电商：浏览记录 → 商品向量空间 → 相似商品推荐

### 重复检测与去重

- 检测相似新闻稿，避免重复推送
- 代码相似度检测，辅助 Code Review 和抄袭检测
- 图片去重，避免重复上传占用存储

### 多模态搜索

- 以图搜图（图片转向量后检索相似图片）
- 以文搜图（使用 CLIP 等模型将文本和图片映射到同一向量空间）
- 语音内容检索（音频转文字后再转向量）

## 常见问题

### Q: 向量数据库和全文搜索（Elasticsearch / Meilisearch）怎么选？

两者适合不同场景，很多系统会**混合使用（Hybrid Search）**：

| 对比项 | 全文搜索 | 向量搜索 |
|--------|----------|----------|
| 匹配方式 | 关键词精确/模糊匹配 | 语义相似度 |
| 同义词处理 | 需要手动配置 | 自动理解 |
| 精确词匹配 | 极强 | 相对弱 |
| 跨语言搜索 | 需要额外配置 | 多语言模型原生支持 |
| 可解释性 | 高（知道为什么匹配） | 低（黑盒） |

混合搜索策略：向量搜索召回语义相关结果，BM25 全文搜索召回精确关键词结果，最后用 RRF（Reciprocal Rank Fusion）融合排序。

### Q: 更换 Embedding 模型需要重建索引吗？

是的，必须重建。不同 Embedding 模型的向量空间完全不同，互不兼容。更换模型时需要对所有数据重新生成向量并重新导入向量数据库。这是选型时要慎重考虑的重要因素。

### Q: 向量维度越高越好吗？

不一定。更高维度通常意味着更强的表达能力，但同时带来：
- 更大的内存和存储占用（线性增长）
- 更慢的查询速度
- 需要更多训练数据才能充分利用高维空间

实践中，384-1536 维对大多数应用场景已经足够，优先根据模型质量而非维度高低来选择。

### Q: 相似度分数的阈值应该设多少？

取决于具体场景和使用的距离度量方式，没有通用标准。建议：
- 先用一批样本数据测试，观察分数分布
- 余弦相似度 > 0.7 通常表示语义相关
- 业务上设置阈值时，宁可过滤掉低相关结果，也不要让无关内容进入 LLM 上下文
