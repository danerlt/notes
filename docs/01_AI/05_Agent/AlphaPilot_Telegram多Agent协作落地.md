# 从 0 到 1 打通 Telegram 多 Agent 协作：AlphaPilot 的设计与落地

## 背景

今天这次工作的目标，不是把一个完整产品一次性做完，而是先把 **AlphaPilot** 的最小可运行协作链路打通：让 Telegram 成为多 Agent 协作的入口，把任务从消息触发、角色分工、执行落盘、阶段汇报，串成一条闭环。

这次实践里，角色被收敛为四个：

- `alpha-pilot`
- `planner`
- `developer`
- `ops`

模型分工也同步收敛：

- `planner -> Claude`
- `developer -> Claude`
- `ops -> Codex`

这次工作的重点，不是追求“全自动自治”，而是先解决两个工程问题：

1. 多 Agent 协作链路怎么设计，才能稳定落地。
2. Telegram 群场景下，消息触发、可见性、权限策略怎么真正跑通。

## 目标

今天这次工作的目标比较明确：

- 建立一套可持续扩展的多 Agent 协作骨架。
- 明确角色职责边界，避免“谁都能做、最后谁都不稳”。
- 在 Telegram 中完成多 bot 接入与 runner bridge 的基础链路。
- 把运行过程中的长期状态落到 artifacts，而不是堆在聊天上下文里。
- 记录并定位群聊接入中的关键策略问题，尤其是 allowlist、`groupPolicy`、`not-allowed`、`no-mention` 这类真实阻塞点。

换句话说，今天不是做“功能大全”，而是做“骨架、路径和约束”。

## 架构设计

### 1. 角色收敛而不是无限扩张

这次没有把 Agent 设计成一组松散的“能力集合”，而是直接收敛成四个工程角色。

- `alpha-pilot` 负责总控与任务编排视角。
- `planner` 负责规划、拆解、形成可执行路径。
- `developer` 负责研发侧执行。
- `ops` 负责运行、接入、桥接、环境联通。

这个设计有两个直接收益：

- 职责边界清晰，消息路由和 handoff 更容易定义。
- 后续无论接模型、接工作流还是接外部入口，都有稳定的角色锚点。

### 2. 模型分工服务于流程，而不是反过来

今天的模型分工采用了相对保守但实用的策略：

- `planner -> Claude`
- `developer -> Claude`
- `ops -> Codex`

原因很直接：规划与开发更适合承担连续推理与实现上下文，运维和接入问题则更适合用 Codex 落地到脚手架、桥接层和运行流程里。这里的关键不是“哪个模型更强”，而是把模型能力映射到稳定的工程责任上。

### 3. 上下文治理优先于“群里全量播报”

这次工作里，一个很重要的设计决定是：**避免上下文爆炸**。

具体做法是：

- 群里只汇报阶段结果，不把全部执行细节持续灌进聊天。
- 长期状态、过程产物、handoff 信息统一沉淀到 `artifacts`。
- 把 Telegram 看成“协作入口 + 状态广播层”，而不是唯一的状态存储层。

这一步非常关键。多 Agent 协作一旦完全依赖聊天上下文，很快就会遇到几个问题：

- 历史信息不可控增长。
- 多 bot 之间难以共享稳定状态。
- 回溯和审计成本极高。
- 任务中断后恢复困难。

所以今天的设计从一开始就把“聊天消息”和“系统状态”分开了。

## 落地步骤

### 1. 先搭知识骨架，再搭执行骨架

这次工作的第一步，不是直接接 Telegram，而是先把项目目录和运行骨架立起来。目录设计围绕“规划、执行、沉淀、回放”展开，而不是简单按代码类型分文件夹。

核心目录包括：

- `spec/plan/worklog`
- `agents`
- `workflows`
- `runs`
- `scripts`
- `docs/agent-orchestration`

这套结构背后的意图很明确：

- `spec/plan/worklog` 用来承接需求、计划、执行日志。
- `agents` 用来定义角色。
- `workflows` 用来定义协作路径和编排规则。
- `runs` 用来承接每次运行实例。
- `scripts` 用来放桥接、触发、辅助执行逻辑。
- `docs/agent-orchestration` 用来记录架构、约束、操作方法与问题复盘。

### 2. 引入 run artifacts、handoff、public summary

在执行层，今天比较重要的落地点有三个：

- `run artifacts`
- `handoff`
- `public summary`

这三个概念分别解决不同问题：

`run artifacts` 负责保存运行过程中的持久化产物，包括阶段结果、状态快照、关键上下文。  
`handoff` 负责在不同角色之间传递足够但不过量的信息。  
`public summary` 负责把对外可见的信息压缩成阶段性汇报，适合发回 Telegram 群。

这也是今天上下文治理的核心做法：**内部状态详尽落盘，对外消息保持克制**。

### 3. 打通 runner bridge

要让 Telegram 真正变成协作入口，中间必须有一层桥。今天落地的关键节点之一，就是 runner bridge。

它承担的是一组非常工程化的职责：

- 接收 Telegram 侧触发。
- 识别应该进入哪个 workflow。
- 把消息映射为一次 run。
- 把执行结果分流到 artifacts、handoff、public summary。
- 决定什么信息应该回流到群里，什么不应该。

这层 bridge 的意义，在于把“聊天平台协议”与“Agent 运行协议”解耦。否则一旦平台行为变化，整个编排层都会被拖着改。

## 关键脚手架

今天这次工作的脚手架，核心不是代码量，而是约束被落成了目录和对象模型。

### 1. 文档与运行双轨并行

`docs/agent-orchestration` 并不是附属品，而是和 `agents`、`workflows`、`runs` 一起构成了系统的一部分。因为多 Agent 系统如果只有运行逻辑，没有显式文档约束，后续一定会出现角色漂移、流程漂移和接入策略漂移。

### 2. spec / plan / worklog 分层

把 `spec`、`plan`、`worklog` 分开，是今天一个很实用的工程决定：

- `spec` 记录要解决什么问题。
- `plan` 记录准备怎么做。
- `worklog` 记录今天实际做了什么、遇到了什么、卡在什么地方。

这能有效避免“最后只剩聊天记录，没人知道系统当前真实状态”。

### 3. runs 作为运行事实源

每次执行进入 `runs`，意味着系统开始有“运行事实源”。后续不管是排障、回放，还是群里只展示 public summary，这个目录都能成为可靠依据。

## Telegram 接入与踩坑

今天最花时间、也最有价值的部分，其实是 Telegram 群场景下的接入调试。

### 1. 多 bot 接入不是把 bot 拉进群就结束

今天确认的一点是：**Telegram 多 bot 接入的真正难点，不在接入动作本身，而在群策略和可见性**。bot 是否在群里，并不等于它一定能稳定收到、识别、响应应该处理的消息。

围绕这一点，今天集中踩到了几类坑：

- 群消息 allowlist
- `groupPolicy`
- `not-allowed`
- `no-mention`
- 群触发与可见性问题
- 群策略调试

这些问题并不是彼此独立的，而是互相耦合。

### 2. allowlist 与 groupPolicy 是第一层门

从今天的过程来看，群接入首先要过的是策略门，而不是执行门。也就是说，在任务有没有被正确分发之前，先要确认消息有没有资格进入系统。

典型表现就是：

- 某些群消息被策略直接拦掉，出现 `not-allowed`。
- 即使 bot 已经在群里，如果群策略不匹配，也可能不会进入预期 workflow。
- `groupPolicy` 如果定义不清，最终表现出来的往往不是“明显报错”，而是“看起来像没触发”。

这类问题特别容易误判成桥接层 bug，实际上常常是准入策略没对齐。

### 3. no-mention 暴露了群触发语义问题

另一个很典型的坑是 `no-mention`。群场景里，消息是否需要显式 mention，决定了 bot 能不能把这条消息当成一个有效触发。

今天这部分暴露出两个现实问题：

- 群里“用户以为已经触发”与“系统认为没有明确触发”之间存在语义鸿沟。
- 即使设计了群触发，也还会受到 bot 可见消息范围的影响。

这意味着 Telegram 群触发不能只靠“经验理解”，必须明确回答几个问题：

- 需要 mention 才触发，还是允许自然语言直接触发？
- 多 bot 同群时，触发优先级怎么区分？
- 可见性不足时，是降级忽略，还是进入明确的拒绝路径？

### 4. 群可见性问题比预期更实战

今天的另一个实际发现是：群触发与可见性问题，不只是“能否收到消息”，还包括“收到的消息是否足够形成稳定上下文”。

这会直接影响两件事：

- 哪些消息能进入 runner bridge。
- public summary 应该如何回流到群里，避免制造更多噪音。

所以今天的结论不是“Telegram 接入已经完全解决”，而是已经识别出最关键的一组约束，并进入针对 `groupPolicy`、allowlist、mention 机制的持续调试阶段。

## 结果

就今天这一次工作而言，已经完成的进展主要有这些：

- AlphaPilot 的角色模型完成收敛，明确为 `alpha-pilot`、`planner`、`developer`、`ops`。
- 模型分工完成初步落位，形成 `planner -> Claude`、`developer -> Claude`、`ops -> Codex` 的职责映射。
- 项目关键目录与脚手架思路已经建立，包括 `spec/plan/worklog`、`agents`、`workflows`、`runs`、`scripts`、`docs/agent-orchestration`。
- run artifacts、handoff、public summary、runner bridge 这些核心对象已经进入设计与落地路径。
- Telegram 多 bot 接入开始进入真实调试阶段，关键阻塞点已经聚焦到 allowlist、`groupPolicy`、`not-allowed`、`no-mention` 以及群触发可见性问题。

同时，也需要诚实说明当前状态：

- 群策略相关问题今天没有被宣称“彻底解决”。
- Telegram 群里的触发语义和可见性边界，仍然是当前最需要继续打磨的部分。
- 现阶段更接近“打通骨架与关键链路”，而不是“所有群场景都稳定可用”。

## 后续计划

接下来的工作重点会比较明确：

- 继续收敛 `groupPolicy`，把群消息准入规则显式化。
- 针对 `not-allowed`、`no-mention` 等场景补齐更可观察的调试信息。
- 进一步验证 Telegram 多 bot 同群时的触发边界与可见性行为。
- 继续强化 artifacts 作为长期状态载体，减少对聊天上下文的依赖。
- 在 runner bridge 上补齐更稳定的输入映射和阶段回传机制。

今天这次工作的价值，不在于“宣布一个完整系统上线”，而在于把 AlphaPilot 从概念推进到了一个真实可调试、可扩展、可复盘的多 Agent 协作骨架上。对于一个要跑进 Telegram 群场景的系统来说，这一步比表面功能更重要。
