# 模型量化

## 什么是模型量化

模型量化（Model Quantization）是将神经网络中的浮点数权重和激活值转换为低精度整数表示的技术。简单来说，就是用更少的比特数来表示模型参数。

### 数值精度类型对比

| 精度类型 | 比特数 | 数值范围 | 内存占用（1B参数） |
|---------|--------|---------|-----------------|
| FP32    | 32 bit | ±3.4×10³⁸ | 4 GB |
| FP16    | 16 bit | ±65504 | 2 GB |
| BF16    | 16 bit | ±3.4×10³⁸ | 2 GB |
| INT8    | 8 bit  | -128 ~ 127 | 1 GB |
| INT4    | 4 bit  | -8 ~ 7 | 0.5 GB |

一个 7B 参数的模型：
- FP32：约 28 GB 显存
- FP16/BF16：约 14 GB 显存
- INT8：约 7 GB 显存
- INT4：约 3.5 GB 显存

### 量化的基本原理

量化的核心思想是将浮点数映射到整数空间：

```python
import torch

def quantize_fp32_to_int8(tensor: torch.Tensor):
    """将 FP32 张量对称量化为 INT8"""
    # 计算缩放因子（对称量化，zero_point = 0）
    x_abs_max = tensor.abs().max()
    scale = x_abs_max / 127.0

    # 量化：除以 scale 后取整并截断到 INT8 范围
    quantized = (tensor / scale).round().clamp(-128, 127).to(torch.int8)
    return quantized, scale

def dequantize_int8_to_fp32(quantized: torch.Tensor, scale: float):
    """将 INT8 张量反量化回 FP32"""
    return quantized.float() * scale

# 示例
original = torch.randn(4, 4)
q, scale = quantize_fp32_to_int8(original)
restored = dequantize_int8_to_fp32(q, scale)

print(f"原始值：{original[0, :4]}")
print(f"量化值：{q[0, :4]}")
print(f"还原值：{restored[0, :4]}")
print(f"最大误差：{(original - restored).abs().max():.6f}")
```

## 量化的好处

### 1. 显存大幅减少

量化后模型可以在更小的 GPU 上运行，甚至在 CPU 上运行：

- Llama3-8B（FP16）：需要 16 GB 显存
- Llama3-8B（INT4）：只需 4-5 GB 显存，消费级显卡即可运行

### 2. 推理速度提升

INT8 矩阵乘法比 FP32 快 2-4 倍，INT4 更快。现代 GPU（如 A100、H100）有专门的 INT8/INT4 计算单元。

### 3. 降低部署成本

- 云端部署可以选用更小的实例
- 边缘设备（手机、嵌入式设备）成为可能
- 同等成本下可以部署更多模型副本

## 量化类型

### PTQ（Post-Training Quantization，训练后量化）

在已训练好的模型上直接进行量化，无需重新训练。

**优点**：
- 快速，不需要训练数据（或只需少量校准数据）
- 实现简单，成本低

**缺点**：
- 精度损失相对较大，尤其是 INT4 以下
- 对某些模型可能有显著性能下降

```python
from transformers import AutoModelForCausalLM, AutoTokenizer
import torch

# 加载原始模型
model_name = "Qwen/Qwen2.5-7B-Instruct"
tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModelForCausalLM.from_pretrained(model_name, torch_dtype=torch.float16)

# PTQ 通常需要少量校准数据来确定量化参数
calibration_texts = [
    "机器学习是人工智能的一个子领域。",
    "The quick brown fox jumps over the lazy dog.",
    "量化技术可以显著减少模型的内存占用。",
    # 推荐使用 128~512 条与实际使用场景相近的文本
]

model.eval()
with torch.no_grad():
    for text in calibration_texts:
        inputs = tokenizer(text, return_tensors="pt")
        outputs = model(**inputs)
        # 量化框架在此过程中收集激活值的统计信息
```

### QAT（Quantization-Aware Training，量化感知训练）

在训练过程中模拟量化操作，让模型学会适应低精度。

**优点**：
- 精度损失最小，效果最好
- 适合对精度要求高的场景

**缺点**：
- 需要训练数据和算力
- 实现复杂，训练时间长（通常比普通训练长 10-30%）

```python
import torch
import torch.quantization

# 准备模型进行 QAT
model.train()

# 设置量化配置（fbgemm 适用于 x86 服务器，qnnpack 适用于 ARM 移动端）
model.qconfig = torch.quantization.get_default_qat_qconfig('fbgemm')

# 插入伪量化节点
torch.quantization.prepare_qat(model, inplace=True)

# 正常的训练循环（伪量化节点会自动模拟量化误差）
for batch in train_dataloader:
    optimizer.zero_grad()
    outputs = model(**batch)
    loss = outputs.loss
    loss.backward()
    optimizer.step()

# 训练完成后转换为真正的量化模型
model.eval()
torch.quantization.convert(model, inplace=True)
```

## 主流量化格式

### GPTQ（GPT Quantization）

专为 LLM 设计的 PTQ 方法，使用二阶信息（Hessian 矩阵）来最小化量化误差。

- 精度：INT4/INT8
- 特点：精度损失小，需要 GPU 推理，速度快（配合 ExLlama2 内核）
- 代表库：`auto-gptq`、`optimum`

### AWQ（Activation-aware Weight Quantization）

基于激活值分布来保护重要权重的量化方法，识别并保留对输出影响最大的权重通道。

- 精度：INT4
- 特点：在 GPTQ 基础上进一步提升精度，对硬件友好
- 代表库：`autoawq`

### GGUF（GPT-Generated Unified Format）

llama.cpp 使用的量化格式，支持 CPU 推理，是 Ollama 的默认格式。

- 精度：Q2_K 到 Q8_0 多种级别
- 特点：跨平台，支持 CPU/GPU 混合推理，无需专用 CUDA 环境
- 代表工具：`llama.cpp`、`Ollama`

常见 GGUF 量化级别（以 7B 模型为例）：

| 格式 | 有效比特数 | 内存占用 | 质量 | 推荐场景 |
|------|-----------|---------|------|---------|
| Q2_K | ~2.5 bit | ~2.7 GB | 较差 | 极度显存受限 |
| Q4_0 | ~4.0 bit | ~4.1 GB | 一般 | 快速测试 |
| Q4_K_M | ~4.8 bit | ~4.8 GB | 良好 | **日常使用推荐** |
| Q5_K_M | ~5.7 bit | ~5.7 GB | 很好 | 质量与速度平衡 |
| Q6_K | ~6.6 bit | ~6.6 GB | 接近原始 | 高质量需求 |
| Q8_0 | ~8.0 bit | ~8.0 GB | 几乎无损 | 最高质量 CPU 推理 |

### BitsAndBytes（BnB）

Hugging Face 集成的量化库，支持 NF4（4-bit Normal Float）格式，常用于 QLoRA 微调。

- 精度：INT8、NF4（4-bit）
- 特点：深度集成到 transformers，配置简单，是 QLoRA 的标准搭档

### 执行量化

```python
from auto_gptq import AutoGPTQForCausalLM, BaseQuantizeConfig
from transformers import AutoTokenizer
import torch

model_name = "Qwen/Qwen2.5-7B-Instruct"
tokenizer = AutoTokenizer.from_pretrained(model_name)

# 配置量化参数
quantize_config = BaseQuantizeConfig(
    bits=4,           # 量化位数（4 或 8）
    group_size=128,   # 分组大小，越小精度越高但内存越大；128 是常用值
    desc_act=False,   # 是否使用激活值排序（True 略慢但精度更高）
    sym=True,         # 对称量化
)

# 加载原始模型（需要足够 GPU 显存）
model = AutoGPTQForCausalLM.from_pretrained(
    model_name,
    quantize_config=quantize_config,
    torch_dtype=torch.float16,
    device_map="auto",
)

# 准备校准数据（128 条样本通常足够）
calibration_dataset = [
    tokenizer(
        "量化是一种通过降低数值精度来压缩模型的技术。",
        return_tensors="pt",
        padding=True,
        truncation=True,
        max_length=512,
    ).input_ids
    for _ in range(128)
]

# 执行量化（7B 模型约需 10-30 分钟，取决于硬件）
print("开始量化，请耐心等待...")
model.quantize(calibration_dataset)

# 保存量化后的模型
save_path = "./qwen2.5-7b-gptq-int4"
model.save_quantized(save_path, use_safetensors=True)
tokenizer.save_pretrained(save_path)
print(f"量化模型已保存至：{save_path}")
```

### INT4（NF4）量化加载

```python
from transformers import AutoModelForCausalLM, AutoTokenizer, BitsAndBytesConfig
import torch

model_name = "Qwen/Qwen2.5-14B-Instruct"

# NF4 量化配置（QLoRA 标准配置）
bnb_config_int4 = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",             # NF4 格式，比普通 INT4 精度更高
    bnb_4bit_compute_dtype=torch.bfloat16, # 反量化后的计算精度
    bnb_4bit_use_double_quant=True,        # 双重量化：对 scale 再次量化，额外节省 ~0.4 bit
)

model = AutoModelForCausalLM.from_pretrained(
    model_name,
    quantization_config=bnb_config_int4,
    device_map="auto",
    trust_remote_code=True,
)
tokenizer = AutoTokenizer.from_pretrained(model_name)

# 构建对话格式并推理
messages = [
    {"role": "system", "content": "你是一个有帮助的 AI 助手。"},
    {"role": "user", "content": "请解释什么是模型量化？"},
]
text = tokenizer.apply_chat_template(messages, tokenize=False, add_generation_prompt=True)
inputs = tokenizer(text, return_tensors="pt").to(model.device)

with torch.no_grad():
    outputs = model.generate(**inputs, max_new_tokens=512, temperature=0.7, do_sample=True)

response = tokenizer.decode(outputs[0][inputs.input_ids.shape[1]:], skip_special_tokens=True)
print(response)
```

### 显存占用对比验证

```python
import torch
from transformers import AutoModelForCausalLM, BitsAndBytesConfig

model_name = "Qwen/Qwen2.5-7B-Instruct"

configs = {
    "FP16（基准）": None,
    "INT8": BitsAndBytesConfig(load_in_8bit=True),
    "INT4 NF4": BitsAndBytesConfig(
        load_in_4bit=True,
        bnb_4bit_quant_type="nf4",
        bnb_4bit_use_double_quant=True,
    ),
}

for name, config in configs.items():
    if torch.cuda.is_available():
        torch.cuda.empty_cache()

    kwargs = {"device_map": "auto"}
    if config is None:
        kwargs["torch_dtype"] = torch.float16
    else:
        kwargs["quantization_config"] = config

    model = AutoModelForCausalLM.from_pretrained(model_name, **kwargs)
    memory_gb = model.get_memory_footprint() / 1e9
    print(f"{name:15s}：{memory_gb:.2f} GB")
    del model
```

## 量化精度对比

### 基准测试参考（Qwen2.5-7B 为例）

| 量化方式 | 困惑度（PPL）↓ | 推理速度（tokens/s）↑ | 显存占用 | 适用场景 |
|---------|--------------|---------------------|---------|---------|
| FP16（基准）| 6.14 | 45 | 14 GB | 高精度生产环境 |
| GPTQ INT8 | 6.17 | 68 | 7.5 GB | GPU 推理，精度优先 |
| GPTQ INT4 | 6.28 | 95 | 4.5 GB | GPU 推理，速度优先 |
| AWQ INT4 | 6.24 | 92 | 4.5 GB | GPU 推理，精度与速度均衡 |
| BnB INT8 | 6.19 | 42 | 8 GB | 微调前加载，快速验证 |
| BnB NF4 | 6.35 | 38 | 5 GB | QLoRA 微调 |
| GGUF Q4_K_M | 6.45 | 25（CPU）| 4.8 GB | 本地 CPU 推理 |
| GGUF Q8_0 | 6.16 | 18（CPU）| 8.0 GB | 高质量 CPU 推理 |

> 困惑度越低越好；数据仅供参考，实际因硬件和任务不同而有所差异。

### 量化方案选择指南

```
需求场景                          推荐方案
─────────────────────────────────────────────────────
GPU 推理，精度优先                 AWQ INT4
GPU 推理，速度优先                 GPTQ INT4 + ExLlama2
CPU 推理，日常使用                 GGUF Q4_K_M
CPU 推理，高质量                   GGUF Q6_K / Q8_0
QLoRA 微调                        BitsAndBytes NF4
生产环境高并发                     vLLM + AWQ/GPTQ
显存极度受限（< 4 GB）             GGUF Q2_K 或 Q3_K_M
```

## 量化对模型性能的影响

### 主要影响方面

1. **复杂推理能力**：数学、逻辑推理对量化误差更敏感，INT4 可能有 2-5% 下降
2. **长文本处理**：量化误差在长序列中累积，超长上下文质量下降更明显
3. **多语言能力**：非英文语言（尤其是低资源语言）受影响相对更大
4. **代码生成**：代码任务精度要求高，建议使用 INT8 以上精度

### 减轻精度损失的方法

```python
from auto_gptq import BaseQuantizeConfig

# 技巧 1：减小 group_size 提升量化粒度（内存换精度）
config_precise = BaseQuantizeConfig(
    bits=4,
    group_size=32,   # 默认 128，改为 32 精度更高但内存占用增加约 15%
)

# 技巧 2：启用激活值排序（desc_act=True）
config_act = BaseQuantizeConfig(
    bits=4,
    group_size=128,
    desc_act=True,   # 对激活值排序后再量化，减少离群值影响
)

# 技巧 3：使用与实际场景匹配的校准数据
# 如果模型主要用于医疗问答，校准数据也应来自医疗领域
def load_domain_calibration_data(tokenizer, domain="medical", n=128):
    texts = load_texts_from_domain(domain)[:n]
    return [
        tokenizer(t, return_tensors="pt", truncation=True, max_length=512).input_ids
        for t in texts
    ]
```

## 常见问题

### Q1：量化会导致模型变慢吗？

不一定。在支持 INT8/INT4 计算的 GPU（NVIDIA Ampere 及更新架构）上，量化通常会加速推理。BitsAndBytes 因需要实时反量化，在某些场景下比 FP16 略慢；而 GPTQ 配合 ExLlama2 内核则明显更快。

### Q2：INT4 量化损失可以接受吗？

对于日常对话、问答类任务，INT4 量化损失通常在 1-3% 以内，完全可以接受。但对于高精度数学计算、专业代码生成，建议使用 INT8 或更高精度。

### Q3：显存不足时加载量化模型报错怎么办？

```python
# 方案：使用 device_map="auto" + max_memory 自动启用 CPU offloading
from transformers import AutoModelForCausalLM, BitsAndBytesConfig

model = AutoModelForCausalLM.from_pretrained(
    "Qwen/Qwen2.5-7B-Instruct",
    quantization_config=BitsAndBytesConfig(load_in_4bit=True),
    device_map="auto",
    max_memory={
        0: "6GiB",       # GPU 0 最多使用 6 GB
        "cpu": "20GiB",  # 其余层放到 CPU 内存
    },
)
```

### Q4：如何快速验证量化后模型质量是否正常？

```python
# 使用几个典型问题快速验证
test_prompts = [

for prompt in test_prompts:

    print(f"Q: {prompt}")
```
