# 大模型微调概述

## 什么是微调（Fine-tuning）

微调（Fine-tuning）是指在一个已经预训练好的大模型基础上，使用特定领域或特定任务的数据进行进一步训练，使模型更好地适应目标任务的过程。

预训练大模型（如 LLaMA、Qwen、Mistral）在海量通用语料上学习了丰富的语言知识和世界知识，但它们并不天然适合特定业务场景。微调就是将这些通用能力迁移到具体任务的桥梁。

```
预训练模型（通用知识）
        ↓
   微调（特定数据）
        ↓
专用模型（领域能力）
```

### 微调与预训练的区别

| 阶段 | 数据量 | 计算成本 | 目的 |
|------|--------|----------|------|
| 预训练 | 万亿 Token | 极高（数百到数千 GPU） | 学习通用语言能力 |
| 微调 | 千到百万条样本 | 中等（1-8 GPU 可完成） | 适配特定任务/风格 |

---

## 为什么需要微调

### 微调 vs Prompt Engineering

Prompt Engineering 通过精心设计输入提示词来引导模型输出，无需修改模型权重，成本极低。但它有明显局限：

- **上下文长度限制**：复杂指令和少样本示例会占用大量上下文窗口
- **推理成本高**：每次请求都需要携带完整的系统提示
- **效果天花板**：对于专业术语、特定风格、私有知识，提示词难以覆盖
- **不稳定**：模型版本更新可能导致提示词失效

### 微调 vs RAG（检索增强生成）

RAG 通过检索外部知识库来补充模型不具备的知识，适合知识更新频繁的场景。

| 对比维度 | 微调 | RAG |
|----------|------|-----|
| 知识更新 | 需要重新训练 | 实时更新知识库 |
| 推理延迟 | 低（无检索步骤） | 较高（需检索） |
| 适合场景 | 风格/行为对齐、私有格式 | 知识密集型问答 |
| 数据隐私 | 知识编码在权重中 | 知识存储在外部 |
| 成本 | 一次性训练成本 | 持续的检索成本 |

**实践中，微调和 RAG 经常结合使用**：先微调让模型学会特定格式和风格，再通过 RAG 补充实时知识。

### 什么时候必须微调

- 需要模型掌握特定**输出格式**（JSON schema、特定模板）
- 需要模型具备特定**领域专业知识**（法律、医疗、金融）
- 需要调整模型的**行为和风格**（语气、回答方式）
- 需要让模型**拒绝特定内容**或**遵循安全规范**
- 需要**压缩推理成本**（小模型微调后可达到大模型效果）

---

## 微调的分类

### 全量微调（Full Fine-tuning）

更新模型的全部参数，效果最好，但资源消耗巨大。

- 一个 7B 模型全量微调至少需要约 80GB GPU 显存（fp16 下）
- 适合有充足 GPU 资源的团队或对效果要求极高的场景

### 参数高效微调（PEFT）

只训练少量参数，冻结大部分预训练权重，大幅降低显存和计算需求。

主流 PEFT 方法：

| 方法 | 原理 | 可训练参数比例 | 特点 |
|------|------|--------------|------|
| **LoRA** | 低秩矩阵分解 | 0.1% - 1% | 最主流，效果与全量接近 |
| **QLoRA** | 量化 + LoRA | 0.1% - 1% | 显存需求极低，消费级 GPU 可用 |
| **Prefix Tuning** | 在输入前添加可训练前缀 | 小于 1% | 适合生成任务 |
| **IA3** | 缩放激活值 | 小于 0.1% | 参数极少，效果略差 |
| **Adapter** | 在层间插入小型网络 | 1% - 5% | 早期方案，已逐渐被 LoRA 取代 |

## 微调的适用场景

### 场景一：指令跟随（Instruction Following）

训练模型按照特定指令格式输出，这是最基础也最常见的微调类型，通常称为 **SFT（Supervised Fine-Tuning）**。

```
输入：请将以下英文翻译成正式中文：Hello, how are you?
输出：您好，请问您近来安好？
```

### 场景二：领域专家化

在特定领域数据上微调，使模型具备专业知识：

- 医疗问答机器人
- 代码补全助手
- 法律文书生成

### 场景三：风格对齐

调整模型的输出风格，使其符合品牌或用户偏好：

- 客服口吻（友好、简洁）
- 学术写作（严谨、规范）
- 创意写作（生动、有趣）

### 场景四：安全对齐（RLHF / DPO）

通过人类反馈或偏好数据训练，使模型输出更符合人类价值观：

- 拒绝有害请求
- 减少幻觉输出
- 提高回答准确性

---

## 微调流程概览

```
1. 确定目标任务
       ↓
2. 准备训练数据（数据收集、清洗、格式化）
       ↓
3. 选择基础模型（模型大小、许可证）
       ↓
4. 选择微调方法（全量 / LoRA / QLoRA）
       ↓
5. 配置训练超参数
       ↓
6. 启动训练（监控 loss、显存）
       ↓
7. 模型评估（验证集指标、人工评测）
       ↓
8. 模型合并与导出
       ↓
9. 部署推理
```

## 主流微调框架对比

### Hugging Face Transformers + PEFT + TRL

最通用的方案，生态最完善。

**优点：**
- 支持几乎所有主流模型
- 文档完善，社区活跃
- 与 HuggingFace Hub 无缝集成
- 灵活可定制

**缺点：**
- 训练速度相对较慢
- 需要手动组装训练流程

```python
from trl import SFTTrainer
from transformers import TrainingArguments

# TRL 的 SFTTrainer 封装了常见的 SFT 训练流程
trainer = SFTTrainer(
    model=model,
    train_dataset=dataset,
    args=TrainingArguments(
        output_dir="./output",
        num_train_epochs=3,
        per_device_train_batch_size=4,
        learning_rate=2e-4,
    ),
)
trainer.train()
```

### Unsloth

专注于训练速度优化的框架，通过自定义 CUDA 内核实现加速。

**优点：**
- 训练速度比原生 Transformers 快 2 - 5 倍
- 显存占用减少约 60%
- API 与 Transformers 兼容，迁移成本低

**缺点：**
- 支持的模型有限（主要是 LLaMA、Mistral、Qwen 系列）
- 免费版不支持多 GPU

```python
from unsloth import FastLanguageModel

# unsloth 的模型加载方式，自动应用优化
model, tokenizer = FastLanguageModel.from_pretrained(
    model_name="unsloth/llama-3-8b-bnb-4bit",
    max_seq_length=2048,
    dtype=None,        # 自动检测
    load_in_4bit=True,
)

# 添加 LoRA 适配器
model = FastLanguageModel.get_peft_model(
    model,
    r=16,
    target_modules=["q_proj", "v_proj", "k_proj", "o_proj"],
    lora_alpha=16,
    lora_dropout=0,
)
```

### LLaMA-Factory

国内最流行的可视化微调框架，支持 Web UI 操作。

**优点：**
- 提供 Web UI，操作门槛低
- 支持数十种模型和微调方法
- 内置数据集管理和模型评估
- 对中文模型（Qwen、ChatGLM）支持好

**缺点：**
- 灵活性相对较低
- 自定义训练逻辑较复杂


## 资源需求估算

### 显存需求（以 LoRA 微调为例）

模型参数的显存占用（fp16）约为参数量 × 2 字节：

| 模型大小 | 模型权重（fp16） | LoRA 梯度+优化器 | 激活值（batch=4） | 推荐显存 |
|----------|----------------|----------------|-----------------|---------|
| 1.5B | 约 3 GB | 约 1 GB | 约 2 GB | 8 GB |
| 7B | 约 14 GB | 约 3 GB | 约 4 GB | 24 GB |
| 13B | 约 26 GB | 约 5 GB | 约 6 GB | 40 GB |
| 70B | 约 140 GB | 约 20 GB | 约 20 GB | 4×80 GB |

**使用 QLoRA（4-bit 量化）可将显存需求降低约 60% - 75%：**

| 模型大小 | QLoRA 所需显存 | 可用消费级 GPU |
|----------|--------------|--------------|
| 7B | 约 6 GB | RTX 3060 12GB |
| 13B | 约 10 GB | RTX 3080 10GB |
| 70B | 约 48 GB | 2×RTX 3090 24GB |

### 训练时间估算

```python
def estimate_training_time(
    num_samples: int,      # 训练样本数
    avg_seq_length: int,   # 平均序列长度（token 数）
    model_size_b: float,   # 模型参数量（十亿）
    gpu_tflops: float,     # GPU 算力（TFLOPS，如 A100=312）
    num_epochs: int = 3,
    efficiency: float = 0.3,  # 实际利用率通常在 20%-40%
) -> float:
    """粗略估算 LoRA 训练时间，返回小时数"""
    # 每个 token 的 FLOPs 约为 6 × 模型参数量（前向+反向）
    flops_per_token = 6 * model_size_b * 1e9
    total_tokens = num_samples * avg_seq_length * num_epochs
    total_flops = flops_per_token * total_tokens
    effective_tflops = gpu_tflops * 1e12 * efficiency
    hours = total_flops / effective_tflops / 3600
    return hours

# 示例：10000 条数据、平均 512 token、7B 模型、A100 GPU
time_h = estimate_training_time(
    num_samples=10000,
    avg_seq_length=512,
    model_size_b=7,
    gpu_tflops=312,
    num_epochs=3,
)
print(f"估算训练时间：{time_h:.1f} 小时")  # 约 0.5 小时
```

## 最佳实践与常见误区

### 最佳实践

1. **从小模型开始验证流程**：先用 1.5B 模型跑通完整流程，再切换到大模型
2. **数据质量优于数量**：1000 条高质量数据往往胜过 100000 条低质量数据
3. **保留基础能力**：加入少量通用对话数据，避免微调后遗忘原有能力（灾难性遗忘）
4. **监控验证集 Loss**：及时发现过拟合，不要只看训练集 Loss
5. **版本管理**：保存每个检查点（checkpoint），方便回滚

### 常见误区

- **误区一**：数据越多越好 → 低质量数据会污染模型，少而精更重要
- **误区二**：训练 Loss 越低越好 → 过拟合会导致模型泛化能力下降
- **误区三**：微调可以替代一切 → 知识时效性问题还是需要 RAG 解决
- **误区四**：直接用大模型生成数据微调 → 需要严格过滤，避免错误传播
